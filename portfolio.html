The Problem

Starting as a new in-house sales representative at a vacation ownership company, I found the onboarding process left significant gaps. The product ecosystem — a tiered vacation ownership credit system with complex VIP benefit structures, multi-step sales processes, and a dense acronym vocabulary — was difficult to internalize quickly. Existing reference materials were fragmented, inaccessible in the field, and offered no way to self-test retention.

I needed a tool I could use on the floor. No such tool existed, so I built one.

What It Does

Tier Up consolidates four categories of reference and practice into a single, tab-navigated web application:

Steps of the Sale — an interactive accordion reference covering each phase of the sales process, with expandable detail for each step.

Quick Reference — a structured glossary of acronyms, post-sale processes, and administrative workflows new reps encounter immediately.

VIP Knowledge Check — an interactive quiz that tests recall of VIP tier benefits, using multiple choice, true/false, and multi-select question formats with immediate feedback.

Objection Handling — a randomized practice tool presenting common owner objections with guided response frameworks and hint support.

Prompt Engineering Approach

Building Tier Up was an iterative prompt engineering exercise from start to finish. Two decisions shaped the quality of the work most significantly.

First, I applied structural thinking from my background in PowerShell and Python scripting — not by writing code, but by thinking in conditionals and hierarchies when composing prompts. Rather than asking Claude to "create a quiz about VIP tiers," I would specify the question type, the answer structure, the feedback behavior on correct and incorrect selections, and the data format the app needed to receive. This precision consistently produced outputs that required less post-processing and fewer correction cycles.

Second, as the app grew across multiple sessions, I developed a practice of including explicit context reminders at the start of new sessions — documenting prior architectural decisions, component structures, and the reasoning behind specific design choices. This approach compensated for the absence of persistent memory in LLMs and allowed me to maintain continuity across a build that spanned weeks. It is documented in detail in the prompt example below.

Prompt Engineering in Practice
Session Continuity Technique
How I maintained coherent AI collaboration across a multi-week build

The Problem: LLMs have no persistent memory

Each new session starts from zero — prior decisions, component structures, and design rationale are invisible to the model.

Early approach — what wasn't working
// Starting a new session cold "Add a new section to the VIP quiz for Gold tier benefits."

Early in the build, resuming work in a new session meant the model had no knowledge of how the app was structured, what naming conventions I had established, how the tab system worked, or what had already been built. The result was outputs that conflicted with existing code — wrong class names, duplicate IDs, incompatible logic — requiring significant rework to integrate.

I recognized the pattern: the quality of the output was directly proportional to how much relevant context I provided at the start of the session. The model wasn't failing — I was under-specifying.

Refined approach — session context block
// Opening a new session with a structured context reminder "We are continuing work on Tier Up, a single-file HTML sales training app. Here is the current structure: - Three tabs: Steps of the Sale, Quick Reference, VIP Study - Tab switching is handled by the showTab() function - Quiz questions are stored in a JavaScript array called quizData, each with: question, type, options, answer, and explanation fields - Accordion sections use the toggleAccordion() function - The active tab button uses the class 'active' - Color scheme: dark background #0d1117, accent blue #1f6feb The last thing we built was the multi-select question type for the VIP quiz. We are now adding Gold tier benefit questions to quizData using the same structure."

By front-loading the session with a structured summary of the app's current state — architecture, naming conventions, the last completed task, and the next task — the model could produce outputs that integrated cleanly on the first attempt. The context block became a standing artifact I updated at the end of each session, so the next session could start immediately from a shared understanding.

This approach reflects a fundamental insight about working with LLMs effectively: the model is only as useful as the context it receives. Treating prompt construction as a system design problem — not just a question-asking exercise — is what separates iterative, productive AI collaboration from frustrating trial and error.

Key insight

Persistent memory is a feature most LLMs don't have — but structured context is a feature any prompt engineer can provide. Recognizing this constraint and designing around it is what allowed a multi-week, multi-session build to maintain coherence. This mental model of how AI systems actually work is directly applicable to any production AI implementation.
